\section{Advanced TLS 1.3 Post-Quantum Integrations}
\label{sec:tls_handshake}
%-----------------------------------------------------------------------

Evaluating an isolated algorithm only shows part of the picture. We need to see how these algorithms handle real network traffic. We solved this by testing the schemes inside the Transport Layer Security (TLS 1.3) protocol \cite{campagna2020tls}. This is the standard security layer for IoT networks.

\subsection{The TLS 1.3 Handshake Framework}
A standard TLS 1.3 handshake uses Elliptic Curve Diffie-Hellman (ECDHE) for the key exchange. It uses ECDSA or RSA to authenticate the server and client. Post-quantum algorithms disrupt this flow. They add massive payloads to the first network trip.

A hybrid ClientHello message must carry two keys. It holds the standard X25519 key (32 bytes) and the new post-quantum public key. For ML-KEM (KYBER)-512, the KeyShare jumps to 832 bytes. This fits easily inside a standard 1,500-byte Ethernet MTU. However, it breaks the 127-byte limit of IEEE 802.15.4 networks. Class 1 sensors rely heavily on this Zigbee/Thread standard.

\subsection{Authentication Fragmentation and Datagram Limitations}
The TLS handshake hits an even larger bottleneck later in the sequence. The Certificate and CertificateVerify messages cause severe network delays.

\begin{itemize}
 \item \textbf{Classical Baseline:} An ECDSA-P256 certificate chain and signature total about 500 to 800 bytes. This payload fits perfectly into a single UDP datagram for DTLS setups.
 \item \textbf{Post-Quantum Overhead:} ML-DSA (DILITHIUM)-2 generates a 1,312-byte public key. It also creates a 2,420-byte signature. The total CertificateVerify payload quickly exceeds 4 KB.
\end{itemize}

This massive size forces protocols like 6LoWPAN to fragment the packets heavily. Sending 4 KB over a 127-byte radio link takes about 40 distinct MAC-layer frames. Real-world IoT networks often experience 5\% packet loss. This loss rate makes it incredibly difficult to receive all 40 frames sequentially. If a router drops just one fragment, the entire handshake fails. The devices must restart the process from scratch. This retry loop drains the sensor's battery life rapidly.

\subsection{The "KEMTLS" Alternative Framework}

To circumvent the signature payload crisis, literature increasingly investigates the KEMTLS framework—a sequence utilizing Key Encapsulation Mechanisms (KEMs) for both ephemeral secrecy \textit{and} explicit authentication, completely stripping heavy digital signatures (ML-DSA / SLH-DSA) from the handshake layer.

KEMTLS authenticates servers by embedding their long-term KEM public key within the certificate. The client generates an ephemeral secret, encapsulates it directly under the server's public key, and transmits the resulting ciphertext. Since \kyber{}-512 ciphertexts are 768 bytes (drastically smaller than a 2,420-byte \dilithium{}-2 signature), KEMTLS circumvents significant datagram fragmentation. 

However, delegating authentication to KEMs shifts the computational burden. The server is forced to execute a heavy Decapsulation operation sequentially simply to authenticate the connection, exposing the infrastructure to highly asymmetrical Denial-of-Service (DoS) attacks where cheap client `ClientHello` connections trigger substantial CPU cycles on backend IoT edge-gateways.

\begin{algorithm}[htbp]
\caption{TLS 1.3 Pure Post-Quantum KEMTLS Handshake}
\label{alg:kemtls}
\begin{algorithmic}[1]
\Require IoT Client $C$, Edge Server $S$ locking Long-Term KEM PK $PK_S$
\Ensure Authenticated Session without Digital Signatures
\State \textbf{[Client Initial Protocol]}
\State $(sk_E, pk_E) \Gets \text{ML-KEM-KeyGen}()$
\State $C \to S: \text{ClientHello} \{pk_E\}$
\State \textbf{[Server Authenticated Response]}
\State $(c_E, SS_E) \Gets \text{ML-KEM-Encaps}(pk_E)$ 
\State $S \to C: \text{ServerHello} \{c_E, \text{Cert}(PK_S)\}$
\State \textbf{[Client Authentication Sequence]}
\State $SS_E \Gets \text{ML-KEM-Decaps}(sk_E, c_E)$
\State $(c_A, SS_A) \Gets \text{ML-KEM-Encaps}(PK_S)$ 
\State $MS \Gets \text{KDF}(SS_E \parallel SS_A)$
\State $C \to S: \{c_A, \text{Finished\_MAC}(MS)\}$
\State \textbf{[Server Validation Sequence]}
\State $SS_A \Gets \text{ML-KEM-Decaps}(SK_S, c_A)$
\State $MS \Gets \text{KDF}(SS_E \parallel SS_A)$
\State Verify $\text{Finished\_MAC}$ dynamically using $MS$
\end{algorithmic}
\end{algorithm}
