\subsection{TOPSIS Decision Framework}

Navigating the PQC transition is a five-dimensional optimization problem. Speed, memory footprint, energy drain, transmission overhead, and cryptographic strength all inherently conflict. To untangle this mathematical knot, we employ TOPSIS \cite{hwang1981multiple}. This technique forcefully collapses disparate metric arrays into a solitary scalar ranking by calculating the geometric distance to theoretical best and worst-case bounds. The resulting scalar ($C_i$) tells a clear story: algorithms clustering near 1.0 are deployment-ready, while those hovering at zero suffer from universal architectural domination.

\begin{algorithm}
\caption{TOPSIS Calculation Routine}\label{alg:topsis}
\begin{algorithmic}[1]
\Require Decision Matrix $X_{m \times n}$, Weights $W_n$
\State \textbf{Norm:} $R_{ij} \Gets X_{ij} / \sqrt{\sum X_{ij}^2}$
\State \textbf{Weight:} $V_{ij} \Gets R_{ij} \cdot W_j$
\State \textbf{Ideals:} Determine $A^+$ and $A^-$ per criterion
\State $D_i^+ \Gets \sqrt{\sum (V_{ij} - V_j^+)^2}$
\State $D_i^- \Gets \sqrt{\sum (V_{ij} - V_j^-)^2}$
\State $\text{Closeness } C_i \Gets \frac{D_i^-}{D_i^+ + D_i^-}$
\State \Return Rank uniquely descending by scalar $C_i$
\end{algorithmic}
\end{algorithm}

\subsection{The Pareto Frontier}

ML-KEM-512 dominates the TOPSIS ranking at NIST Level 1 ($C_i = 0.9452$ on Class 2, Table XVI) and remains competitive even against Level 3 alternatives. Stepping from Level 1 to Level 5 within the ML-KEM family roughly doubles both timing and memory (Table IX: 2.10$\times$ and 1.95$\times$ respectively). ML-DSA shows steeper scaling: 2.45$\times$ timing and 2.15$\times$ memory from Level 2 to Level 5. For Class 1 hardware, this scaling pushes ML-DSA-3 and above into OOM territory entirely.

ML-KEM-768 occupies the sweet spot for deployments that need Level 3 security without the OOM risk — it completes in 12.85 ms on Class 2 and 72.85 ms on Class 1, well within typical DTLS timeout windows.

\subsection{Significant Differences in Architectures}

The ANOVA F-test returned $p < 0.001$ for all four metrics — key generation, encapsulation, decapsulation, and peak memory — with effect sizes ($\eta^2$) between 0.75 and 0.91 (Table \ref{tab:anova}). Algorithm family, not parameter level, is the dominant factor. Tukey's HSD post-hoc test (Table \ref{tab:tukey_hsd}) confirms that the only statistically significant pairwise gap at $p < 0.05$ is between ML-DSA-2 and SLH-DSA-128f, where the mean difference exceeds 499 seconds. Within the ML-KEM family, differences of 10-12 ms between adjacent security levels are not significant after Bonferroni correction.

\subsection{Hybrid Classical-Quantum Scheme Dynamics}

Lattice-based assumptions lack the multi-decade track record of RSA or ECC, so NIST recommends hybrid schemes that run classical and post-quantum key exchanges in parallel, concatenating both shared secrets through a KDF. The cost is steep. Table \ref{tab:hybrid_overhead} quantifies it: ECDHE + ML-KEM-512 takes 9.4 ms and 16.5 KB RAM on Class 2, roughly $2.2\times$ the pure classical baseline. Adding ML-DSA for authentication pushes RAM to 55.4 KB — already exceeding the Class 1 ceiling. Hybrid modes are viable only on Class 2 hardware with Ethernet-class MTUs ($\geq$ 1500 bytes); on 802.15.4 links, the combined 1,632-byte KEM payload alone demands 20 MAC-layer frames.

\begin{algorithm}[htbp]
\caption{Standardized Hybrid ECDHE + ML-KEM Exchange}
\label{alg:hybrid}
\begin{algorithmic}[1]
\Require Client $C$, Server $S$
\Ensure Quantum-Resistant Master Secret $MS_{hybrid}$
\State \textbf{[Client Execution]}
\State $(ek_c, P_c) \Gets \text{ECDHE-KeyGen}()$
\State $(sk_{pq}, pk_{pq}) \Gets \text{ML-KEM-KeyGen}()$
\State $C \to S: \{P_c, pk_{pq}\}$
\State \textbf{[Server Execution]}
\State $(ek_s, P_s) \Gets \text{ECDHE-KeyGen}()$
\State $SS_{cl} \Gets \text{ECDHE-Multiply}(ek_s, P_c)$
\State $(c, SS_{pq}) \Gets \text{ML-KEM-Encaps}(pk_{pq})$
\State $MS_{hybrid} \Gets \text{KDF}(SS_{cl} \parallel SS_{pq})$
\State $S \to C: \{P_s, c\}$
\State \textbf{[Client Verification]}
\State $SS_{cl} \Gets \text{ECDHE-Multiply}(ek_c, P_s)$
\State $SS_{pq} \Gets \text{ML-KEM-Decaps}(sk_{pq}, c)$
\State $MS_{hybrid} \Gets \text{KDF}(SS_{cl} \parallel SS_{pq})$
\end{algorithmic}
\end{algorithm}
